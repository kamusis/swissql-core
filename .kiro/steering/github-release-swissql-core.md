# GitHub Release Workflow — SwissQL Core

## Tag Convention

Two independent tag prefixes, each triggers its own CI workflow:

| Tag format | CI workflow | Artifact |
|---|---|---|
| `cli-vX.Y.Z` | `release-cli.yml` | Cross-platform binaries → GitHub Release |
| `backend-vX.Y.Z` | `release-backend-image.yml` | Docker image reference → GitHub Release |

Both tags are always bumped together to the same version number.

## Version Bump Rules (Conventional Commits)

Scan commits since the last tag, scoped by component directory:
- CLI bump: commits touching `swissql-cli/`
- Backend bump: commits touching `swissql-backend/`

| Bump | Trigger |
|---|---|
| **major** | `BREAKING CHANGE` in body, or `type(scope)!:` / `type!:` subject |
| **minor** | Any `feat(...):` commit |
| **patch** | Everything else (fix, docs, chore, refactor, perf, test) |

## Tagging Steps

### 1. Confirm main is clean

```bash
git status && git log --oneline -5
```

### 2. Find the latest tags

```bash
git tag -l | grep -E "^(cli|backend)-v" | sort -V | tail -5
```

### 3. Determine next version

Run these two commands to see all relevant commits since the last tag (replace `TAG` with the latest tag found above, e.g. `backend-v0.2.1`):

```bash
# All component commits since last tag
git log --pretty=format:"%s" TAG..HEAD -- swissql-cli swissql-backend
```

Apply bump rules:
- Any `feat:` or `feat(...):` → **minor** (`X.Y+1.0`)
- Any `BREAKING CHANGE` or `!:` → **major** (`X+1.0.0`)
- Otherwise → **patch** (`X.Y.Z+1`)

### 4. Tag and push

```bash
git tag -a cli-vX.Y.Z -m "CLI: <summary of CLI commits>."
git tag -a backend-vX.Y.Z -m "Backend: <summary of backend commits>."
git push origin cli-vX.Y.Z backend-vX.Y.Z
```

### 5. Verify

```bash
git show cli-vX.Y.Z --quiet
git show backend-vX.Y.Z --quiet
```

## Version Semantics

- **Major (X.0.0)**: breaking API changes, incompatible profile storage format, removal of supported DB types
- **Minor (X.Y.0)**: new CLI commands, new backend endpoints, new driver support, new credential reference types
- **Patch (X.Y.Z)**: bug fixes, error handling, docs, performance, minor UX

## Rules

- Never tag directly on a feature branch — tags go on `main` after merge
- Tag message format: `CLI: <commit subjects>.` / `Backend: <commit subjects>.`
- Do not manually edit GitHub Release notes generated by CI
- If only one component changed, still bump both tags to keep versions in sync
- No latest tag yet? Use `0.0.0` as base and scan the last 50 commits: `git log --pretty=format:"%s" -50 -- swissql-cli swissql-backend`
